# 地图与房间属性修改指南（Metroidvania-System-master）

## 1. 现有地图是怎么管理的

这个项目里“地图/房间”主要分成两层：

- **逻辑地图（MetSys 数据库）**：用于小地图、房间网格、房间连通、房间与场景的绑定。核心在 [MapData.gd](file:///D:/Task_Panel/0_AliceJOH/0_AOJ_Reference/workspace/Metroidvania-System-master/addons/MetroidvaniaSystem/Scripts/MapData.gd) 与 Settings 的 `map_data_file`（默认 `res://MapData.txt`，见 [Settings.gd](file:///D:/Task_Panel/0_AliceJOH/0_AOJ_Reference/workspace/Metroidvania-System-master/addons/MetroidvaniaSystem/Scripts/Settings.gd#L13-L18)）。
- **运行时房间（Godot 场景 .tscn）**：真正承载 TileMap、碰撞、物件等。项目房间场景在 `CoreEngine/Maps/*.tscn`，例如 [Corridor.tscn](file:///D:/Task_Panel/0_AliceJOH/0_AOJ_Reference/workspace/Metroidvania-System-master/CoreEngine/Maps/Corridor.tscn)。

房间场景与 MetSys 数据库的“绑定”靠每个房间场景里挂一个 `RoomInstance`：

- `RoomInstance` 会在编辑器/运行时根据 `owner.scene_file_path` 去 MetSys 的 MapData 找“这个场景对应哪些 cell / 属于哪个 room”（见 [RoomInstance.gd](file:///D:/Task_Panel/0_AliceJOH/0_AOJ_Reference/workspace/Metroidvania-System-master/addons/MetroidvaniaSystem/Scripts/RoomInstance.gd#L53-L69)）。

运行时的加载与切房间：

- 主游戏脚本 [Game.gd](file:///D:/Task_Panel/0_AliceJOH/0_AOJ_Reference/workspace/Metroidvania-System-master/CoreEngine/Scripts/Systems/Game.gd) 继承 [MetSysGame.gd](file:///D:/Task_Panel/0_AliceJOH/0_AOJ_Reference/workspace/Metroidvania-System-master/addons/MetroidvaniaSystem/Template/Scripts/MetSysGame.gd)。
- `MetSysGame.load_room(path)` 负责卸载旧 map、实例化新 map、加到场景树，并发出 `room_loaded`（见 [MetSysGame.gd](file:///D:/Task_Panel/0_AliceJOH/0_AOJ_Reference/workspace/Metroidvania-System-master/addons/MetroidvaniaSystem/Template/Scripts/MetSysGame.gd#L38-L55)）。
- 切房间过渡由模块 [AreaRoomTransitions.gd](file:///D:/Task_Panel/0_AliceJOH/0_AOJ_Reference/workspace/Metroidvania-System-master/CoreEngine/Scripts/Systems/AreaRoomTransitions.gd) 监听 `MetSys.room_changed` 并调用 `game.load_room()`。

## 2. 现在如何“灵活地修改到具体某个地图的某个房间的若干个属性”

项目原先的 Actor 架构里，已经能在“房间加载完成”这个时机做额外逻辑（`Game.init_room()` 会向 Workbench 广播 `TYPE_ROOM_LOADED`，见 [Game.gd](file:///D:/Task_Panel/0_AliceJOH/0_AOJ_Reference/workspace/Metroidvania-System-master/CoreEngine/Scripts/Systems/Game.gd#L436-L448)）。

为了能**按“房间场景路径”定向改属性**，我补全了一个“房间属性覆写”流程：

- **RoomPropertyCatalog**：用一个 GDScript 字典声明“某个房间要应用哪些修改”（见 [RoomPropertyCatalog.gd](file:///D:/Task_Panel/0_AliceJOH/0_AOJ_Reference/workspace/Metroidvania-System-master/CoreEngine/Scripts/World/RoomPropertyCatalog.gd)）。
- **RoomPropertyManagerActor**：监听 `TYPE_ROOM_LOADED`，在每次加载房间后，把对应的 ops 应用到当前 `game.map` 上（见 [RoomPropertyManagerActor.gd](file:///D:/Task_Panel/0_AliceJOH/0_AOJ_Reference/workspace/Metroidvania-System-master/CoreEngine/Scripts/Actor/RoomPropertyManagerActor.gd)）。
- 在 [Workbench.gd](file:///D:/Task_Panel/0_AliceJOH/0_AOJ_Reference/workspace/Metroidvania-System-master/CoreEngine/Scripts/Core/Workbench.gd) 的 `_ready()` 里注册了这个 Actor，所以开箱即用。

这样你就可以：

- 以“房间场景路径（res://.../xxx.tscn）”为 key
- 列出一组 ops（对具体节点做“换贴图 / 改判定 / 改位置 / 改 TileMap 的砖块风格”等）

## 3. RoomPropertyCatalog 的写法与常用操作

入口文件：

- [RoomPropertyCatalog.gd](file:///D:/Task_Panel/0_AliceJOH/0_AOJ_Reference/workspace/Metroidvania-System-master/CoreEngine/Scripts/World/RoomPropertyCatalog.gd)

结构：

```gdscript
static func get_room_properties() -> Dictionary:
	return {
		"res://CoreEngine/Maps/SomeRoom.tscn": {
			"ops": [
				{ "op": "...", "path": "...", ... },
			]
		},
	}
```

### 3.1 修改任意节点属性：set_props

适合：改位置、缩放、可见性、数值参数等（只要是该节点真实存在的 property）。

```gdscript
{
	"op": "set_props",
	"path": "SomeNode2D",
	"props": {
		"position": Vector2(300, 200),
		"visible": true
	}
}
```

`path` 规则：

- `path` 是相对当前房间根节点 `game.map` 的 NodePath
- 用 `.` 代表房间根节点本身

### 3.2 修改资源引用：set_resources

适合：换 Sprite2D.texture、TextureRect.texture、TileMapLayer.tile_set（不推荐整体替换，通常只换 source/tile 更合理）等。

```gdscript
{
	"op": "set_resources",
	"path": "FloatingPlatform/Sprite2D",
	"props": {
		"texture": "res://CoreEngine/Sprites/Tiles/DarkTiles.png"
	}
}
```

### 3.3 修改 CollisionShape2D 的 shape：set_shape

适合：改悬空地面/机关的判定范围（CollisionShape2D）。

```gdscript
{
	"op": "set_shape",
	"path": "FloatingPlatform/CollisionShape2D",
	"shape": {
		"class": "RectangleShape2D",
		"size": Vector2(220, 20)
	}
}
```

只要是 `ClassDB` 可实例化的 Shape2D 类型（如 `RectangleShape2D`、`CircleShape2D` 等）都可以。

### 3.4 TileMapLayer：整层换“砖块风格”（换 source_id）：replace_source_id

适合：把某个房间地面/墙壁贴图整体换成另一套 tilesheet（前提：两套 TileSet source 的 atlas 坐标兼容）。

这个项目的 `TileSet` 有多个 atlas source（例如 `Tiles.png` / `DarkTiles.png` / `MirageTiles.png`，见 [Tileset.tres](file:///D:/Task_Panel/0_AliceJOH/0_AOJ_Reference/workspace/Metroidvania-System-master/CoreEngine/Resources/Tileset.tres#L551-L556)）。

示例（已内置在 Catalog 里，用 Corridor 房间把 source 1 -> 2）：

```gdscript
{
	"op": "replace_source_id",
	"path": "TileMap/Foreground",
	"from": 1,
	"to": 2
}
```

注意：`Corridor.tscn` 的背景层节点名写成了 `Bakcground`（拼写如此），所以示例里用的是 `TileMap/Bakcground`（见 [RoomPropertyCatalog.gd](file:///D:/Task_Panel/0_AliceJOH/0_AOJ_Reference/workspace/Metroidvania-System-master/CoreEngine/Scripts/World/RoomPropertyCatalog.gd)）。

### 3.5 TileMapLayer：精确替换某一种 tile：replace_tile

适合：只替换特定“悬空地面砖”的皮肤/判定（通过换到另一张 tile，或换 alternative_tile）。

```gdscript
{
	"op": "replace_tile",
	"path": "TileMap/Foreground",
	"from": { "source_id": 1, "atlas_coords": Vector2i(3, 6), "alternative_tile": 0 },
	"to": { "source_id": 2, "atlas_coords": Vector2i(3, 6), "alternative_tile": 0 }
}
```

如果你想改“判定范围”，TileMap 的碰撞来自 TileSet 的 physics layer。做法一般是：

- 在 TileSet 里做一个碰撞不同的 tile 变体（或 alternative_tile）
- 然后用 `replace_tile` 把旧 tile 替换成新 tile

## 4. 如何定位“具体某个房间、某个对象”的 path

推荐做法：

- 打开目标房间场景（`CoreEngine/Maps/xxx.tscn`）
- 确认要改的节点在场景树里的相对路径，例如 `TileMap/Foreground`、`FloatingPlatform/CollisionShape2D`
- 把这个相对路径写到 op 的 `path` 里
