# 游戏事件管理架构提案（战斗 / 跑图 / 对话 / 主线一体触发）

## 目标

- 让战斗、地图、对话、主线脚本都能互相触发
- 让事件管理与物理地图解耦，避免“地图节点改动就破坏主线触发”
- 提供可视化编辑入口，统一维护触发与动作链

## 核心抽象

- 事件节点（EventNode）
  - 语义单位：一个可被触发执行的业务片段
  - 关键字段：`Id`、`Domain`、`Scope`、`Triggers`、`Actions`
- 触发规则（TriggerRule）
  - 描述“什么信号触发此事件”
  - 典型类型：`EnterMap`、`BattleWin`、`DialogueEnd`、`Signal`、`VariableChanged`
- 分发动作（DispatchAction）
  - 描述“事件触发后要做什么”
  - 典型动作：`StartEvent`、`StartDialogue`、`StartCombat`、`ChangeMap`、`SetVariable`

## 解耦策略

- 地图只负责发信号，不直接编排主线
  - 例如地图层发 `EnterMap(map_id)`、`ReachPortal(portal_id)` 事件
- 事件系统订阅信号并执行触发匹配
  - 匹配规则在事件数据层维护，不写死在地图脚本
- 对话、战斗、剧情推进也统一走事件总线
  - 对话结束发 `DialogueEnd(dialogue_id)`
  - 战斗结算发 `BattleWin/ BattleLose(combat_id)`
  - 再由事件图决定后续流转

## 运行时建议

- 事件总线（EventBus）
  - 输入：来自地图/战斗/对话/系统的标准化信号
  - 输出：分发到事件引擎
- 事件引擎（EventEngine）
  - 根据触发规则筛选可执行事件
  - 按优先级、冷却、一次性标记执行
  - 执行动作并产生新信号
- 状态存储（EventStateStore）
  - 保存变量、事件完成状态、冷却时间戳
  - 写入存档层，支持继续游戏恢复

## 工具落地

- 事件编辑器工程：
  - `GodotTools/EventStudio/EventStudio`
- 编译产物目录：
  - `GodotTools-Build/EventStudio`
- 数据文件格式：
  - `*.events.json`（编辑态）
  - `*.runtime.events.json`（运行时扁平图）
- 编辑器内置校验：
  - 校验 `StartEventId` 是否存在
  - 校验事件 `Id` 唯一性
  - 校验 `StartEvent` 动作目标是否存在
  - 错误阻断保存/导出，警告可确认后继续

## 接入建议

- 地图模块：只发信号，不写流程
- 战斗模块：只发结算信号，不直接切图
- 对话模块：只发对话生命周期信号，不直接改主线变量
- 主线推进：全部通过 `StartEvent` 动作串联

这样可保证：地图结构变化时，只需改 scope 或触发条件，不需要重写剧情逻辑。

## 当前已落地的运行时接入

- Workbench 已注册运行时事件流 Actor：`CoreEngine/Scripts/Actor/RuntimeEventFlowActor.gd`
- 已支持监听并转译：
  - `room_loaded`（地图）
  - `area_loaded`（区域）
  - `level_event`（关卡事件）
  - `runtime_event_signal`（通用信号）
- 已支持动作执行：
  - `StartEvent`
  - `ChangeMap`
  - `SetVariable`
  - `EmitSignal`
  - `StartDialogue` / `StartCombat` / `CompleteQuest` / `CustomScript`（通过 `level_event_request` 转发）
- 新游戏流程中会发出：
  - `runtime_event_start`（触发 `startEventId`）
  - `runtime_event_signal: Game.StartNew`
- 对话系统会发出：
  - `runtime_event_signal: Dialogue.Start`
  - `runtime_event_signal: Dialogue.End`
